# Roo Code - Gemini Thinking Mode: Actionable Implementation Plan

**Project Goal:** Integrate Google Gemini's "thinking" capabilities (`includeThoughts` and `thinkingBudget`) into a fork of the Roo Code VS Code extension, focusing on minimal, merge-friendly modifications.

**Report Reference:** This plan is based on the "Integrating Gemini's Advanced Reasoning into Roo Code" technical blueprint.

**Guiding Principles for Implementation:**
1.  **Minimal Core Changes:** Prioritize additive changes and leverage existing abstractions. Avoid deep refactoring of core Roo Code components.
2.  **Opt-In Feature:** "Thinking Mode" should be off by default or activated via specific Custom Mode configurations.
3.  **Leverage Existing Structures:** Utilize Roo Code's Custom Mode system, AI provider abstractions, and webview messaging.
4.  **Clear Separation:** Isolate "thinking"-related logic as much as possible.
5.  **Mergeability:** Structure changes to be easily reviewable and mergeable into the main Roo Code repository.

## Phase 1: Core API Integration & Basic UI (Target: 2-3 Sprints)

### Sprint 1: Backend Foundation & API Adaptation

**Task 1.1: Fork Roo Code Repository & Setup Development Environment**
* **Action:** Fork the `roocodeinc/roo-code` repository on GitHub. Clone the fork locally.
* **Action:** Set up the development environment as per Roo Code's `CONTRIBUTING.md` or developer documentation. Ensure build processes and tests run successfully.
* **Files:** `README.md`, `CONTRIBUTING.md`, `package.json` (for build scripts).
* **Agent Actionable:** Create a new branch (e.g., `feature/gemini-thinking-mode`).

**Task 1.2: Define Core Types & Interfaces for Thinking Mode**
* **Action:** Define TypeScript interfaces for `ThinkingConfig` (to include `includeThoughts` and `thinkingBudget`) and for messages that represent "thoughts."
* **Files:**
    * Likely in `src/exports/types.ts` or a new `src/modules/thinking/types.ts`.
    * Example:
        ```typescript
        // In a relevant types file
        export interface GeminiThinkingConfig {
          includeThoughts?: boolean;
          thinkingBudget?: number; // 0 to 24576 or undefined
        }

        export interface ThinkingMessagePart {
          type: 'thought';
          taskId: string; // Associate with the current task
          content: string;
          timestamp: number;
        }
        ```
* **Agent Actionable:** Create/update type definition files.

**Task 1.3: Modify AI Service Layer for `thinkingConfig`**
* **Goal:** Adapt the service responsible for making LLM calls to accept and use the new `GeminiThinkingConfig`.
* **Action:** Identify the core AI interaction service (e.g., a class or set of functions in `src/services/ai/` or `src/core/llmProvider.ts` - actual file name needs to be located in Roo Code codebase).
* **Action:** Modify the function(s) that construct the API request payload for Gemini models.
    * Add an optional `thinkingConfig?: GeminiThinkingConfig` parameter to these functions.
    * If `thinkingConfig` is provided and the target model is a compatible Gemini model (e.g., `gemini-2.5-flash-preview-05-20`), append the `thinkingConfig` (properly formatted for the Google AI SDK) to the `generationConfig` in the API request.
* **Minimal Change Strategy:** Ensure that if `thinkingConfig` is not provided, the API call proceeds as before.
* **Files:** The main AI service file(s) making calls to Gemini.
* **Agent Actionable:** Update the identified AI service file(s). Add unit tests for constructing requests with and without thinking config.

**Task 1.4: Extend Custom Mode Schema & Parsing**
* **Goal:** Allow Custom Modes to specify thinking parameters.
* **Action:** Update the JSON/YAML schema documentation for Custom Modes to include optional `geminiThinkingConfig` (or similar, like `thinkingIncludeThoughts: true/false`, `thinkingBudget: number|"auto"|"off"`).
* **Action:** Modify the Custom Mode loading/parsing logic (e.g., in a hypothetical `src/core/CustomModeManager.ts` or `src/modes/ModeService.ts`) to read these new optional parameters.
* **Action:** Store these parameters within the loaded mode definition object.
* **Minimal Change Strategy:** New parameters should be optional. If not present in a mode's definition, thinking features remain disabled or use a sensible default (e.g., off).
* **Files:** Custom mode definition schema (if explicit), custom mode parsing/loading service.
* **Agent Actionable:** Update mode parsing logic and relevant type definitions for mode configurations. Add unit tests for loading modes with and without thinking parameters.

### Sprint 2: Processing Thoughts & Basic Webview Integration

**Task 2.1: Process API Responses for Thoughts (Extension Side)**
* **Goal:** Extract thought summaries from Gemini API responses and differentiate them from the final answer.
* **Action:** In the AI service layer (modified in Task 1.3), adapt the response handling logic for Gemini models.
    * When `includeThoughts` was true in the request:
        * Iterate through the response parts (as per Gemini SDK documentation).
        * If a part is a "thought" (`part.thought === true`), create a `ThinkingMessagePart` (defined in Task 1.2).
        * If a part is the final answer, process it as a regular AI response.
* **Action:** Implement logic to send these distinct `ThinkingMessagePart` objects and regular response messages to the webview via the existing message passing mechanism (e.g., `webviewMessageHandler.ts`). Introduce a new message type for thoughts (e.g., `POST_MESSAGE_TYPE.ADD_THOUGHT_MESSAGE`).
* **Files:** AI service file(s), `src/core/webview/webviewMessageHandler.ts` (or equivalent).
* **Agent Actionable:** Update response handling and message posting logic. Add unit tests for parsing responses containing thoughts.

**Task 2.2: Basic Webview State Management for Thoughts**
* **Goal:** Enable the webview to store and distinguish thought messages.
* **Action:** In the webview's state management (e.g., Zustand store or React Context used by `webview-ui/src/components/chat/ChatView.tsx`):
    * Update the message type definition to include a `type: 'thought'` or similar discriminator.
    * Add a new action/reducer to handle incoming `ADD_THOUGHT_MESSAGE` from the extension and add `ThinkingMessagePart` objects to the chat message list.
* **Files:** Webview state management files (e.g., store, actions, reducers), `webview-ui/src/types/messageTypes.ts` (or equivalent).
* **Agent Actionable:** Update webview state logic.

**Task 2.3: Basic UI for Displaying Thoughts (Non-Collapsible)**
* **Goal:** Display thought messages in the chat UI in a visually distinct but simple manner.
* **Action:** Modify `webview-ui/src/components/chat/ChatRow.tsx` (or the component responsible for rendering individual messages).
    * Add conditional rendering logic: If a message is of type "thought," render it using a distinct style (e.g., different background color, prefix like "[Thinking...] ").
    * For this initial phase, thoughts can be displayed as simple text blocks.
* **Action (Optional but Recommended):** Consider adapting the existing `webview-ui/src/components/chat/ReasoningBlock.tsx` if its structure is suitable for displaying thought summaries with minimal changes. It might need props to differentiate between "tool reasoning" and "Gemini thoughts."
* **Files:** `webview-ui/src/components/chat/ChatRow.tsx`, potentially `ReasoningBlock.tsx`, relevant CSS/Tailwind utility classes.
* **Agent Actionable:** Update `ChatRow.tsx` to render thought messages.

**Task 2.4: Pilot "Thinking" in One Custom Mode**
* **Goal:** Test the end-to-end flow with a single Custom Mode.
* **Action:** Create or modify one existing Custom Mode (e.g., a copy of "Architect Mode") in its YAML/JSON definition:
    * Set `geminiThinkingConfig.includeThoughts: true`.
    * Set `geminiThinkingConfig.thinkingBudget` to a moderate value (e.g., 1024) if using Gemini Flash, or leave it unset for Gemini Pro.
    * Add a simple instruction to its system prompt like: "When responding, first provide your thought process as a thought summary."
* **Action:** Manually test this mode by sending a complex prompt and observing if thoughts are requested, received, and displayed in the webview.
* **Files:** The YAML/JSON file for the chosen pilot Custom Mode.
* **Agent Actionable:** Configure and test the pilot mode.

### Sprint 3: Streaming, Error Handling & Basic Settings

**Task 3.1: Implement Streaming for Thought Summaries**
* **Goal:** Improve perceived performance by streaming thoughts to the UI as they are generated.
* **Action (Extension Side):** Modify the AI service to handle streaming responses from Gemini when `includeThoughts` is true.
    * As chunks of data arrive, identify if they are thought parts or final answer parts.
    * Send these chunks to the webview incrementally using the message passing system.
* **Action (Webview Side):** Modify `webview-ui/src/components/chat/ChatView.tsx` and its state management to append incoming streamed thought/response chunks to the appropriate message in the UI.
* **Minimal Change Strategy:** This will likely require more significant changes to message handling on both sides. Focus on adapting existing streaming logic if Roo Code already supports streaming for regular responses.
* **Files:** AI service files, `webviewMessageHandler.ts`, `ChatView.tsx`, webview state management.
* **Agent Actionable:** Implement streaming logic on both extension and webview sides.

**Task 3.2: Basic Error Handling for Thinking Features**
* **Goal:** Gracefully handle common errors related to thinking mode.
* **Action (Extension Side):** In the AI service:
    * Implement basic error handling for API calls involving `thinkingConfig` (e.g., log errors, send an error message to the webview if thinking request fails).
    * Specifically handle `429 RESOURCE_EXHAUSTED` errors with a simple retry or by informing the user.
    * Log `response.usage_metadata.thoughts_token_count` if available.
* **Files:** AI service files.
* **Agent Actionable:** Add error handling and logging.

**Task 3.3: Basic Global Setting for Thinking (Developer/Experimental)**
* **Goal:** Provide a simple way to enable/disable thinking for testing, without full UI integration yet.
* **Action (Extension Side):** Add an experimental boolean configuration setting in `package.json` (under `contributes.configuration`) for globally enabling `includeThoughts` for Gemini models (e.g., `rooCode.gemini.experimental.enableThinking`).
* **Action (Extension Side):** Read this setting in the AI service and use it as a default if a Custom Mode doesn't specify `includeThoughts`.
* **Files:** `package.json`, AI service files, configuration reading logic (e.g., in `src/core/ConfigurationManager.ts`).
* **Agent Actionable:** Add and use the experimental configuration setting.

## Phase 2 (Partial): Enhanced UI & Budget Management (Focus on Minimal Viable Enhancements)

### Sprint 4: UI Refinements & Initial Budget Control

**Task 4.1: Collapsible Thought Blocks in UI**
* **Goal:** Improve UI clarity by making thought blocks collapsible.
* **Action (Webview Side):** Modify `webview-ui/src/components/chat/ChatRow.tsx` (or the `ThinkingBlock.tsx` if created/adapted):
    * Implement expand/collapse functionality for thought messages.
    * Thoughts should ideally be collapsed by default.
    * Store the collapsed/expanded state locally in the component or in the webview's message state.
* **Files:** `ChatRow.tsx` (or `ThinkingBlock.tsx`), relevant CSS/Tailwind.
* **Agent Actionable:** Implement collapsible UI for thoughts.

**Task 4.2: UI for `thinkingBudget` in Custom Mode Settings (if applicable)**
* **Goal:** Allow users to configure `thinkingBudget` if they are editing Custom Mode JSON/YAML directly. (Full UI in `SettingsView.tsx` is deferred).
* **Action:** No direct code changes, but update documentation for Custom Modes to explain the new `thinkingBudget` field and its accepted values (`"auto"`, `"off"`, `<number>`).
* **Files:** Documentation for Custom Modes.
* **Agent Actionable:** Update documentation.

**Task 4.3: Basic `thinkingBudget` Logic in AI Service**
* **Goal:** Ensure the AI service correctly interprets `thinkingBudget` values from Custom Modes.
* **Action (Extension Side):** In the AI service, when preparing `thinkingConfig`:
    * If `thinkingBudget` from Custom Mode is a number, use it.
    * If `"off"` or `0`, set `thinking_budget: 0` (or omit if it disables thinking by default).
    * If `"auto"` or undefined, do not set `thinking_budget` (let Gemini Flash use its default dynamic budget).
* **Files:** AI service files.
* **Agent Actionable:** Implement logic for interpreting `thinkingBudget` settings.

## Testing Strategy (Throughout All Phases)

* **Unit Tests:**
    * Test new type definitions.
    * Test construction of API requests with/without `thinkingConfig`.
    * Test parsing of API responses containing thoughts.
    * Test Custom Mode parsing for new thinking parameters.
    * Test webview state updates for thought messages.
* **Integration Tests (Extension Side):**
    * Test the flow from a Custom Mode definition to an API call with correct `thinkingConfig`.
    * Test message passing of thoughts from extension to webview.
* **Manual Testing (VS Code Extension):**
    * Thoroughly test with Gemini 2.5 Flash and Gemini 2.5 Pro (if Pro access is available).
    * Test different `thinkingBudget` values.
    * Test streaming of thoughts.
    * Test various Custom Modes with thinking enabled/disabled.
    * Verify UI rendering and collapsibility of thoughts.
    * Test error scenarios (API errors, rate limits).

## Pull Request Strategy (for merging to main Roo Code)

1.  **Small, Focused PRs if Possible:** If the work can be broken down logically (e.g., backend API changes first, then UI changes), consider multiple smaller PRs. However, for a cohesive feature like this, a single well-structured PR might be better.
2.  **Clear PR Description:**
    * Explain the "Why": The value proposition of Gemini Thinking Mode for Roo Code users.
    * Explain the "What": A summary of changes made, highlighting the minimal-invasive approach.
    * Explain the "How": Overview of the implementation (API changes, Custom Mode integration, UI display).
    * Link to the original "Integrating Gemini's Advanced Reasoning into Roo Code" report for context.
3.  **Comprehensive Testing:** Emphasize that the changes have been tested and do not break existing functionality. Include summaries of test cases.
4.  **Opt-In by Default:** Ensure the feature is off by default and only activates when explicitly configured in Custom Modes or via (initially experimental) settings.
5.  **Code Comments:** Add clear comments to new/modified code sections explaining the logic for thinking features.
6.  **Address Potential Concerns:** Proactively address potential concerns like increased token usage and latency, and explain how the implementation provides controls (e.g., via Custom Modes, `thinkingBudget`).

This plan provides a structured approach to integrating Gemini's thinking mode into Roo Code with a strong emphasis on maintainability and mergeability. The actual file names and specific class/function names within Roo Code will need to be identified by the AI agent during implementation by analyzing the provided codebase.
